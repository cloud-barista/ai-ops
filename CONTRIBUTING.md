# Contributing

Welcome to the AI service-control prototype repository.

This project focuses on a 1st-year Go-based functional prototype for AI LLM
operation management, agent registration management, and AI application
deployment/control strategy validation.

Thank you for your interest in contributing to this project. This document provides a simple guide for contributing to our repository.

## Contribution Types

You can contribute in several ways:

- Open an issue
  - Bug report
  - Feature request
  - Enhancement suggestion
  - Question about the project

- Open a Pull Request
  - Documentation update
  - Source code improvement
  - Bug fix
  - Test code addition
  - Refactoring

Contributions are not limited to the examples above. Any helpful improvement is welcome.

## Ground Rules and Expectations

Before contributing, please keep the following rules in mind:

- Be respectful and constructive.
- Keep each contribution focused on one clear purpose.
- Make sure the code runs correctly before submitting.
- Write clear commit messages and Pull Request descriptions.
- Share important logs and error messages when they are related to your contribution.

Please read the [Contributor Code of Conduct](./CODE_OF_CONDUCT.md). By participating in this project, you agree to abide by its terms.

## Development Guide

This project follows the development direction below:

- Go language development is required.
- At least two LLM-based coding agents should be used for cross-checking.
- Development should focus on framework prompts and shared prompt documents.
- Logs and error messages should be written clearly and sufficiently.
- Coding agents can be used for implementation, but humans should review, test, and verify the results.

## Contribution Guide

Before contributing, please check existing issues and Pull Requests.

If your contribution is minor, such as fixing a typo or improving documentation, you can directly open a Pull Request.

If your contribution is major, such as adding a new feature, changing the project structure, or modifying the main workflow, please open an issue first so the team can discuss it.

For Pull Request details, include the change summary, validation commands,
expected reviewer notes, and any relevant logs directly in the Pull Request
description.

## Pull Request

When opening a Pull Request, please include:

- What you changed
- Why you changed it
- How you tested it
- Any notes for reviewers

If reviewers request changes, update the same branch and push again. The Pull Request will be updated automatically.

## Documentation

Project outputs are managed through GitHub.

Main documents may include:

- Requirement specification
- Function/API guide
- Installation and usage guide
- Test guide

Documentation should be concise and practical.

## Questions

If you are unsure about the contribution process, please open an issue or ask the project maintainer before making a large change.

We appreciate every contribution that helps improve this project.
