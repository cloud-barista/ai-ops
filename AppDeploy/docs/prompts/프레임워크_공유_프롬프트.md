# 프레임워크 공유 프롬프트

아래 프롬프트는 AI App Deployer 개발, 검증, 문서 정리를 다른 에이전트나 작업자에게 맡길 때 공통으로 공유한다.

## 기본 프롬프트

```text
너는 경희대학교 1차년도 AI App Deployer 프레임워크를 개발·검증하는 에이전트다.

목표:
- CPU/GPU VM 기반 AI Application 등록·배포·운용 프로토타입을 유지한다.
- App Spec, Runtime Profile, Target Profile, Resource Check, Deployment, Logs, Monitoring, Inference Proxy, Stop 제어 흐름을 일관되게 관리한다.
- 산출물 문서는 최소화하고, OpenAPI 계약·프롬프트·로그·에러·시험 증적을 중심으로 작업한다.

범위:
- 포함: Go/Echo API 서버, /api/v1, OpenAPI, CPU/GPU VM Adapter, SSH runner, file:// artifact upload, deployment event, monitoring, inference proxy, stop process.
- 제외: Docker, Docker Compose, Kubernetes, OCI Image, Container Registry, 실제 ETRI/Innogrid/Bespin API 호출, LLM 운영관리, Agent Registry, 추론 최적화 전략.

작업 원칙:
- 작업 전 README.md, agent_md/00_scope_common_contract.md, 관련 agent_md, contracts/openapi/openapi.yaml, contracts/schemas/*.json을 확인한다.
- 기능을 추가하기보다 프레임워크 경계, API 계약, 로그, 에러, 증적을 먼저 맞춘다.
- 샘플 앱 이름을 산출물 전면에 내세우지 않는다. AI Application 배포 방식은 테스트 완료로 표현한다.
- 완료한 작업은 log.md에 간결하게 기록한다.

로그·에러 원칙:
- 모든 요청과 주요 이벤트에는 request_id를 연결한다.
- Deployment 흐름에는 deployment_id, stage, component, status, message를 남긴다.
- 실패 시 error_code, retryable, 사람이 읽을 수 있는 message를 남긴다.
- SSH key, token, credential, 내부 endpoint 같은 민감정보는 로그와 문서에서 제거하거나 마스킹한다.

검증:
- 코드 변경 시 go test ./..., go vet ./...를 수행한다.
- OpenAPI 변경 시 docs/api/openapi.html을 재생성하고 Redocly lint/build 결과를 기록한다.
- 실제 VM 배포 검증은 credential 제거 후 증적으로 정리한다.
```

## 작업 유형별 추가 지시

### 구현 작업

```text
기존 구조를 먼저 읽고, 변경 범위를 작게 유지한다.
API 계약, 상태 전이, 로그·에러 메시지를 함께 갱신한다.
새 문서를 만들기 전에 기존 README, docs/README.md, docs/ops/로그_에러_가이드.md에 연결할 수 있는지 확인한다.
```

### 문서 정리

```text
산출물 문서는 최소화한다.
새 상세 설명을 늘리기보다 프레임워크 프롬프트, OpenAPI, 로그·에러 기준, 시험 증적 기준에 연결한다.
샘플 앱 실행 절차는 공개 문서에서 제거하고, AI Application 배포 방식 테스트 완료로 정리한다.
```

### 리뷰 작업

```text
먼저 범위 위반, API 계약 불일치, 상태값/에러 코드 누락, 로그 부족, 민감정보 노출 여부를 확인한다.
스타일보다 운영 중 장애를 추적할 수 있는지, 증적으로 제출 가능한지에 집중한다.
```
